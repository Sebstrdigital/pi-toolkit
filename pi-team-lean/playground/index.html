pi-team-lean — flow

Per-story pipeline. Two LLM calls; everything else is git, bash, exit codes.
deterministic — code, git, bash, exit codes
LLM — pi subprocess, prompt-driven
I/O — read/write to disk or repo state

Bootstrap (once per pi-team-lean run)

B1
I/O
Parse sprint.json from arg
src/index.ts → readFileSync(sprintPath) → JSON.parse → Sprint
No transformation. Sprint is TAKT-compatible subset: { base_branch?, staging_branch?, test_command?, worker_model?, qa_model?, stories[] }.
B2
DET
Verify clean working tree
src/git.ts → ensureCleanTree() → git status --porcelain
Refuse to start if anything is uncommitted. No exceptions.
B3
DET
Cut staging branch from base
src/git.ts → cutBranch(staging, base) → git checkout base; git checkout -b staging
Defaults: base = main, staging = pi-team-lean/staging-<timestamp>. If staging already exists, resume on it (no recut).
B4
I/O
Initialize .pi-team-lean/sprint-state.json
src/index.ts → mkdirSync acceptance/ + writeFileSync state
All stories seeded as pending. Persisted after every state change so TAKT retro can read partial runs.
B5
DET
Topological sort stories by depends_on
src/index.ts → topoSort()
Pure function. Detects cycles, throws on unknown dependency ids. No LLM input on ordering.

Per-story loop (sequential, one at a time)

S0
DET
Check unmet dependencies
src/index.ts → blockers = depends_on.filter(d ≠ "merged")
If any blocker is not merged, mark story skipped with reason and continue to next story.
If skipped → next story (no LLM, no branch ops).
S1
LLM
qa-author drafts acceptance/<story-id>.sh
src/pi.ts → spawn pi -p --no-session --mode text [--model qa_model]
prompt: src/prompts.ts → qaAuthorPrompt(story)
Once per story (cached on disk; reused on retry). Output is raw bash — script must exit 0 only when story is correctly implemented. Build-side never sees this script.
If qa-author fails or empty → story marked failed, no branch cut.
S2
DET
Cut feature branch from staging
src/git.ts → cutBranch(pi-team-lean/<staging>/story-<id>, staging)
Per-story isolation. Worker only ever sees its own branch.
S3
LLM
worker implements + commits
src/pi.ts → spawn pi -p --no-session --mode text [--model worker_model]
prompt: src/prompts.ts → workerPrompt(story, scope, test_command)
Worker is told to read project conventions (CLAUDE.md, .pi/AGENTS.md), implement, run tests itself for tight feedback, then git add -A && git commit. Worker has no merge/branch authority.
If worker exits non-zero → back to staging, story failed.
S4
DET
Verify worker actually committed
src/git.ts → commitsOnBranch(feature, staging) → git log staging..feature --format=%H
Empty list = worker exited successfully but produced no commits. Treat as failure — never silently merge a no-op branch.
If no commits → back to staging, story failed.
S5
DET
Run test_command
src/index.ts → runTestCommand(test_command, repoCwd)
Defaults to npm test; per-story override via story.test_command. Pass = exit 0. Captures stdout+stderr for failure trail.
If test_command fails → back to staging, story failed with last 1.5KB of output.
S6
DET
Run acceptance/<id>.sh
src/index.ts → runTestCommand(bash <script>, repoCwd)
QA gate. Bash assertions written by S1 are evaluated against the implemented branch. Pass = exit 0. The implementation cannot read or game these assertions because the worker prompt never included them.
If acceptance fails → back to staging, story failed.
S7
DET
Merge feature → staging (--no-ff)
src/git.ts → mergeNoFf(feature, staging, "merge: <id> <title>")
No fast-forward keeps the per-story branch boundary visible in history. Story state → merged.
On merge → next story.

Teardown (after last story)

T1
I/O
Final state write + checkout staging
src/index.ts → writeFileSync(state) + checkout(staging)
Print summary line per story. Staging branch left intact for human review or PR. main is never touched by this harness.

What was bootstrapped

Built from scratch

  • pi-team-lean/ — new package
  • src/index.ts — main loop, state machine, sprint runner
  • src/git.ts — thin git wrapper (44 LOC)
  • src/pi.ts — pi subprocess wrapper (46 LOC)
  • src/prompts.ts — qa-author + worker prompt templates
  • src/types.ts — Sprint / Story / SprintState shapes
  • bin/pi-team-lean.mjs — CLI entry
  • package.json, tsconfig.json, README.md

Defaults / extensions used

  • No pi extensions. Spawns the pi CLI as a subprocess only.
  • Standard Node 22 builtins: child_process, fs, path.
  • Devdeps: typescript, @types/node. No runtime deps.
  • Reuses your existing pi install + auth (host subscription).
No skills, no agents, no MCP, no extension API touched. The harness is a plain Node CLI.

Counts

Per story

  • LLM calls: 2 (qa-author once, worker once)
  • Deterministic steps: 6 (S0, S2, S4, S5, S6, S7)
  • I/O writes to state: ≥4 (after every status change)

Per sprint

  • Bootstrap: 5 steps (3 DET + 2 I/O)
  • Teardown: 1 step (I/O)
  • LLM calls total: 2 × N stories
